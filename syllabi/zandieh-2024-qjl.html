Syllabus — TurboQuant and the rotation-based quantization line

Summary

QJL — “1-Bit Quantized JL Transform for KV Cache Quantization with Zero Overhead” (Zandieh, Daliri, Han — AAAI 2025) — quantizes a vector $x \in \mathbb{R}^d$ to $\text{sign}(Sx)$ where $S$ is a random Johnson–Lindenstrauss matrix, giving a 1-bit-per-output-dimension code with no per-block quantization constants stored. The estimator for inner products is asymmetric: apply QJL (sign-bit) to one vector and a plain (unquantized) JL to the other; the result is an unbiased estimator of the original inner product with variance $\frac{\pi}{2m}\|x\|_2^2\|y\|_2^2$ for $m$ output dims. Eliminating the per-block scales is what gives QJL its “zero overhead” — standard 3-bit KV cache methods burn 1–2 bits of metadata per block on scale + zero-point storage. On Llama-2 / Llama-3 with various NLP tasks, QJL compresses KV caches >5× at 3 bits with no accuracy loss and faster decode runtime. QJL is the direct building block reused as TurboQuant’s residual stage.

Key points

The construction

  1. Sample a random matrix $S \in \mathbb{R}^{m \times d}$ with i.i.d. Gaussian entries (or sub-Gaussian; structured Hadamard works too).
  2. QJL side (the “stored” vector): compute $h(x) = \text{sign}(Sx)$ — 1 bit per row.
  3. JL side (the “query” vector): compute $Sy$ in full precision (no sign).
  4. Estimator: $\hat{\langle x, y\rangle} = \frac{\pi/2}{m} \|x\|_2 \cdot (Sy)^T h(x)$ — unbiased for $\langle x, y\rangle$.

The $\sqrt{\pi/2}$ factor compensates for the bias the sign operation would otherwise introduce. The $\|x\|_2$ scalar is the only per-vector overhead.

Why “zero overhead”

Conventional 3-bit quantization stores per-block scales and zero-points at higher precision (e.g., FP16) — this metadata costs 1–2 bits per quantized value, which is why “3-bit quantization” effective rate is closer to 4–5 bits. QJL stores only a single $\|x\|_2$ per vector (amortized across hundreds of coordinates), so the effective rate is essentially the raw bit count. This is the “zero overhead” claim.

Asymmetric estimator

Quantizing both sides ($\text{sign}(Sx)^T \text{sign}(Sy)$) gives a biased estimator (the arcsine kernel — angular distance, not inner product). The asymmetric design — quantize one side, keep the other full-precision — preserves the inner-product geometry. In KV-cache use this matches the access pattern perfectly: keys are stored quantized; queries arrive at decode time in full precision.

KV cache results

Across Llama-2-7B, Llama-3-8B, and Mistral-7B on LongBench / GSM8K / NIAH: - 3-bit QJL ≈ FP16 quality (negligible perplexity / score deltas) - >5× memory reduction on the KV cache - Faster end-to-end decode because attention becomes memory-bound and the smaller KV cache fits more of the active context in cache hierarchy

How QJL differs from KIVI / KVQuant

  • KIVI / KVQuant: data-dependent — per-channel or per-token scales computed at write time, stored as metadata, used at read time.
  • QJL: data-oblivious — the random matrix $S$ is fixed at model load; nothing per-block to compute.
  • Trade-off: KIVI/KVQuant can squeeze more accuracy out of the data distribution; QJL is cheaper and trivially streaming.

Limitations

  • The output dimension $m$ for QJL is typically $\geq d$ (often $m = 2d$ or $m = 3d$) to hit the desired variance — so “1 bit per output coordinate” doesn’t mean “1 bit per input coordinate”. Effective compression is closer to 3 bits/input-coord at $m = 3d$.
  • The asymmetric pattern requires you to know in advance which side is “stored” vs “queried” — fine for KV cache, awkward for symmetric similarity search.

TurboQuant — implementation landscape

State (as of May 2026)

No official Google Research implementation is publicly available. Google announced TurboQuant in a research.google blog and published the paper (arXiv:2504.19874) but has not released code on github.com/google-research/ at the time of writing. The implementation landscape is entirely community-driven, which means: lots of choice, no single canonical reference, varying fidelity to the paper.

If you need to understand the algorithm, read the paper plus the official QJL repo (which implements the second-stage primitive verbatim). If you need to use it in inference, pick the community implementation closest to your runtime.

Official QJL implementation (the building block)

amirzandieh/QJL — Amir Zandieh is a TurboQuant co-author and the QJL paper’s first author. This is the most authoritative code for the 1-bit JL residual stage that TurboQuant reuses. Useful as ground truth when evaluating any TurboQuant implementation.

What to verify before trusting an implementation

The paper’s claims are sensitive to several construction details that are easy to get wrong:

  • Rotation source: a uniform random orthogonal matrix (or a structured proxy: random Hadamard with random sign flips). Block-Hadamard without sign-flip diagonal is not equivalent and degrades quality.
  • Lloyd-Max codebook: must be computed for the Beta marginal at dimension $d$, not for a Gaussian approximation. The asymptotic equivalence holds at large $d$ but the exact $d=128$ table differs.
  • Residual normalization: $\gamma = \|r\|_2$ must be stored and used in the QJL stage’s decode formula. Skipping it (or storing $\|x\|_2$ instead) breaks unbiasedness.
  • Mixed-precision channel selection: the 2.5-b/channel result uses 32 outlier channels at 3 bits + 96 at 2 bits, with the outlier channels selected based on… what? The paper underspecifies this; different implementations make different choices.

Open questions for the implementation track

  • Will Google release official code? If not, which community fork becomes de facto canonical?
  • Triton vs CUDA: 0xSero/turboquant is Triton-only; a hand-tuned CUDA kernel could close the gap to FP16 decode latency further.
  • vLLM mainline integration: still a draft as of May 2026.
  • Integration with KV-cache eviction (StreamingLLM, H2O): orthogonal, but the composition isn’t validated.

KV cache quantization

What the KV cache is

In a transformer decoder, each attention layer at decode step $t$ needs the keys $K_{1..t}$ and values $V_{1..t}$ for every prior token to compute attention from the current query. Instead of recomputing them, they are cached in GPU memory and grown by one row per generated token. This is the KV cache.

Size per token, per layer, per head: $2 \cdot d_{\text{head}}$ floats. For Llama-3-8B (32 layers, 8 KV heads, 128 d_head, FP16): ~131 KB per token. A 128k-token context uses ~16 GB of KV cache — much larger than the 16 GB model itself.

Why quantizing it is worthwhile

Two reasons, both about memory:

  1. Capacity: serving longer contexts and larger batches is bottlenecked by KV cache, not model weights (weights are shared across requests; cache is per-request).
  2. Bandwidth: in autoregressive decode, attention is memory-bound — each token reads the whole cache. Reducing bits per cache entry directly reduces wall-clock decode latency.

Even a “modest” 4× compression unlocks 4× context length at the same memory budget, or 4× batch size, or correspondingly faster decode.

What makes KV cache quantization different from weight quantization

Weight quantization (GPTQ, AWQ, QuaRot) KV cache quantization
Calibration Offline, on a small held-out set Must be online — new tokens arrive at decode
Distribution shift Static (weights don’t change) Constantly drifting per-request
Read pattern Dense (whole weight matrix) Sparse, growing (one query × $t$ keys)
Per-block metadata cost Amortized over many forward passes Paid per generated token

The “online + no calibration” requirement is why data-oblivious schemes (QJL, TurboQuant) are particularly natural here, whereas data-dependent schemes (KIVI, KVQuant) need careful streaming statistics.

Outlier structure: keys vs values

A consistent empirical finding (KIVI, KVQuant, SmoothQuant):

  • Keys: persistent outliers on specific channel indices (~3–5% of channels), across all tokens. Per-channel scales help.
  • Values: outliers concentrate on specific tokens (e.g., punctuation, BOS), across all channels. Per-token scales help.

This asymmetry is what motivates KIVI’s split scheme. Rotation-based methods sidestep the issue: a random rotation flattens both kinds of outlier into well-behaved Beta-distributed coordinates, so per-coordinate quantization works uniformly.

The state of the art (mid-2025 / early-2026)

Three rough families:

  • Per-block scalar with smart partitioning: KIVI (per-channel K, per-token V), KVQuant (per-channel K, pre-RoPE, sparse outlier handling). Excellent at 2–3 bits but pay 1–2 bits of metadata overhead.
  • Rotation + scalar: QuaRot, SpinQuant for weights; QJL, TurboQuant for KV cache. Data-oblivious, zero metadata, near-Shannon-rate.
  • Sparse + dense hybrids: KVQuant’s dense-sparse split, attention-aware eviction (H2O, StreamingLLM) — orthogonal to quantization, sometimes stacked.

Headline numbers: parity with FP16 at 3.5 b/channel (TurboQuant), small degradation at 2 b/channel (KIVI, QJL). The race is now to push below 2 bits without crossing the quality cliff.

Where the cliff is

For Llama-class models on long-context tasks: - $\geq 4$ b/channel: indistinguishable from FP16 - 3.5 b/channel: parity (TurboQuant claim) - 2.5–3 b/channel: small but measurable degradation on hard tasks (LongBench Reasoning, NIAH at extreme depth) - $< 2$ b/channel: open problem; quality falls off for everyone

RAG vs persistent agent memory

The confusion

The term “memory” gets used for two related-but-distinct things:

  1. Retrieval over a corpus — at query time, embed the query, kNN-search a vector store of pre-indexed documents, stuff the top-k into the prompt. This is RAG (Lewis et al., 2020).
  2. Persistent state of an interacting agent — the agent has accumulated facts, preferences, and inferences over many sessions, and brings them to bear on the current turn.

Both involve fetching past content. They are often conflated because they share the same machinery (embeddings, vector DBs, kNN). But the design constraints are different in ways that matter.

A side-by-side

RAG Agent memory
Corpus mostly static (documents you indexed once) grows continuously with every interaction
Distribution fixed at index time drifts per user / per session
Write pattern batch streaming
Forgetting no concept essential (contradictions, stale facts)
Consolidation none reflection, summarization, graph linking
Update primitives insert (and rarely delete) ADD, UPDATE, DELETE, NOOP
Failure mode retrieve irrelevant chunk persist a wrong fact, or fail to retire a stale one
Granularity document chunks (paragraphs) facts, episodes, reflections

Why this distinction matters for AI agents

Treating agent memory as “just RAG over conversation history” is the implicit baseline in most LLM chat systems — and it is exactly what every modern agent memory paper argues is insufficient. The reasons:

  1. No write-time intelligence. RAG indexes whatever you give it; agent memory has to decide what is worth remembering (importance scoring in Generative Agents (Park et al., 2023)) and whether it contradicts an existing fact (UPDATE in Mem0 (Chhikara et al., 2025)).
  2. No consolidation. RAG cannot synthesize a higher-level inference from many chunks. Generative Agents and Mem0 both invest heavily in reflection: an LLM-driven step that reads many memories and writes new abstracted ones back.
  3. No bounded state. RAG stores everything; agent memory has to fit useful history into a working budget — whether that’s a working-context cap (MemGPT), a forgetting curve (Generative Agents), or a fixed-size state matrix (Delta MEM (Lei et al., 2026)).
  4. Online distribution shift. A long conversation drifts; the embedding of “what the user wants” at turn 100 may not match its embedding at turn 1. RAG indexes are static; agent memory must adapt.

RAG-as-memory is a baseline, not a solution

In every paper in this syllabus, BM25 RAG or embedding RAG is the baseline; the contribution is what gets added on top:

  • MemGPT adds a structured tier hierarchy and self-managed function calls.
  • Generative Agents adds importance scoring, recency decay, and reflection.
  • Mem0 adds explicit ADD/UPDATE/DELETE/NOOP semantics and a graph variant.
  • Memorizing Transformers keeps the kNN but moves it inside the model’s representational space (KV pairs, not text).
  • Titans / δ-mem abandon retrieval entirely in favor of a continuously-updated parametric state.

When RAG-as-memory is fine

  • The user is asking about documents (e.g., a knowledge base assistant). Then it really is a retrieval problem.
  • The conversation is short and the LLM context fits the whole history.
  • You have a separate mechanism (a structured profile, a CRM) that holds the persistent facts; the LLM is stateless on top.

When RAG-as-memory breaks

  • The conversation is long enough that critical facts have left the context (multi-session assistants).
  • Facts mutate over time (“my address used to be X, now it’s Y” — vanilla RAG happily retrieves both).
  • The agent needs to reason over episodes, not chunks (“what did we discuss last Tuesday?”).
  • Tasks require the agent to recognize that a piece of context is important enough to remember, vs. transient.

These are the regimes where the agent-memory literature lives.

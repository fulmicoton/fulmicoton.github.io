Syllabus — Memory for AI agents

Summary

MemGPT (Packer, Wooders, Lin, Fang, Patil, Stoica, Gonzalez — UC Berkeley, 2023) frames the LLM context window as RAM and proposes that the LLM should manage its own memory hierarchy via function calls — a virtual-memory OS for language models. The system has three tiers: main context (the in-prompt window, fast but capped), recall storage (recent message history, paged in on demand), and archival storage (long-term, unbounded vector store). The LLM emits function calls to read, write, and search the lower tiers; an interrupt-driven control loop pages content in and out without violating the context cap. Evaluated on document analysis far exceeding GPT-4’s 32k window and on multi-session conversational agents (MSC dataset), MemGPT shows that explicit, self-managed memory substantially outperforms vanilla long-context prompting on tasks requiring retention across sessions. The paper became the reference for textual memory in the agent-memory taxonomy and led to Letta — the leading open-source stateful-agent framework.

Key points

The OS analogy

The central pitch is rhetorical but operationally clean: traditional OS virtual memory creates the illusion of large memory through paging between RAM and disk. MemGPT does the same for an LLM:

OS concept MemGPT analog
RAM Main context (LLM input window)
Disk Recall + archival storage (external)
Page table LLM-emitted function calls
Interrupt “Heartbeat” / control-loop tick that re-invokes the LLM
Kernel Outer control loop (Python) that executes function calls

Memory hierarchy

  • Main context — the actual prompt seen by the LLM. Split into:
    • System instructions (fixed)
    • Working context (free-form scratchpad the LLM can rewrite — typically holds the user persona and a few key facts)
    • FIFO queue (recent messages, capped, oldest get evicted to recall)
  • Recall storage — full history of past messages and function calls (vector-indexed). Searchable.
  • Archival storage — out-of-conversation knowledge: long documents, persistent facts. Vector store.

Self-managed via function calls

The LLM is taught (via system prompt) to emit functions like: - core_memory_append(name, content) — write to the working context (the in-prompt scratchpad). - core_memory_replace(name, old, new) — edit working context. - conversation_search(query) — query recall storage. - archival_memory_insert(content), archival_memory_search(query) — write/read archival. - send_message(...) — actually reply to the user (otherwise the loop continues).

The LLM is its own memory manager: it decides what to promote (write to working context) and what to demote (let it fall off the FIFO into recall).

Interrupt-driven control loop

After every LLM turn, the outer loop checks for emitted function calls and runs them, then re-invokes the LLM. The LLM can chain multiple function calls without a user response (it just emits another request_heartbeat instead of send_message). This is how multi-step memory operations (e.g., “search archival, then update working context, then reply”) are stitched together within one logical user turn.

Key experiments

  • Document QA on long PDFs / multi-doc — MemGPT substantially outperforms truncated-context baselines on GPT-4 and GPT-3.5-Turbo. Notable because the LLM has to actively search archival, not just be given the right chunk.
  • Multi-session chat (MSC) — MemGPT remembers facts across sessions where vanilla 32k-context chat models forget.
  • Nested-key retrieval — synthetic stress test (find a key hidden many layers deep across documents). MemGPT scales while context-only baselines collapse.

Why it landed

  • First serious production-shaped memory framework: it’s not a research artifact, it’s an architecture for stateful agents.
  • The OS framing made the design choices teachable and the abstractions reusable (working context / recall / archival is now standard vocabulary).
  • Open-sourced from day one; the project rebranded to Letta in 2024 and became one of the dominant open-source agent frameworks.

Limitations the field is still grappling with

  • Function-call reliability — the LLM has to choose to manage its memory. Weaker LLMs forget to write to working context, or write too aggressively.
  • Compression is text-based — the working context summarizes in natural language, paying tokenization cost and accumulating compaction loss (Laban 2025, Hong 2025).
  • Retrieval noise — archival search returns approximate matches; irrelevant chunks pollute the prompt.
  • No graded forgetting — the FIFO eviction is binary; there is no notion of an item slowly decaying or being consolidated.

These limitations are exactly what Mem0, Generative Agents (with reflection), and the latent-memory line (Titans, δ-mem) try to address.
